## 2. 設計判断軸

このプロジェクトでは以下の 4 軸で設計判断を行う。論点で迷ったときは、この軸に立ち返って判断する。

| 軸 | 内容 | 確立 Day |
|---|---|---|
| 軸1 | 可逆な判断には YAGNI、不可逆な判断には先回り設計 | Day 2 |
| 軸2 | 判断は LLM、計算は Python(関心の分離の LLM 時代版) | Day 3 |
| 軸3 | Defense in Depth(各層が独立に防御、全層同時にすり抜ける確率を実用上ゼロまで下げる) | Day 3 |
| 軸4 | Few-shot は「Phase の幅」と「status の書き方の対比」を示す教材 | Day 5 |

### 軸4 の詳細(Day 5 で新規確立)

Few-shot の主目的は次の 3 つ:
1. **出力フォーマットの見本**(JSON 構造、evidence の書き方)
2. **status 判定の基準**(confirmed / partial / unconfirmed / risk の使い分け)
3. **Phase 判定の境界線**(Phase 2 と Phase 4 の対比で、中間 Phase も推測可能に)

副次的目的(ハルシネーション抑制)は v1 ではプロンプトのみで対応し、Few-shot に過剰な期待をしない(Day 3 論点 5C)。

論点 A〜E の判断は、すべて軸4 に照らして「対比を成立させるか」「対比を邪魔しないか」で評価した。

## 3. 共通設定

### 自社(売り手)

| 項目 | 内容 |

## 4. Phase 2 案件仕様

### 4.1 案件サマリ

| 項目 | 内容 |
|---|---|
| 顧客企業 | Tier 2 自動車部品メーカー(中堅、年商 200〜500 億規模) |
| 取扱製品 | エンジン周辺部品(例: ベアリング、シール、ガスケット) |
| 案件規模 | 年間 ¥8M(税抜)、1 拠点導入想定 |
| 商談回数 | 2 回目(初回顔合わせ後の本格ヒアリング) |
| 商談形式 | 顧客先で対面、所要 60 分 |
| 議事録形式 | 録音文字起こし、5,000〜10,000 字 |

### 4.2 登場人物

#### 顧客側 2 名

**田中 部長(製造部 部長)**
- 45 歳、現場たたき上げ
- Champion 候補(発話で明確に「課題を解決したい」意思を示す)
- 性格: 本音を出すタイプ、感情表現がストレート
- 立ち位置: 商談の主導者、社内の推進役になりうる人物

**鈴木 課長(生産管理 課長)**
- 38 歳、データに強い
- Pain 言語化役(具体的な業務フロー・失敗事例を語る)
- 性格: 論理的、数字で話す
- 立ち位置: 田中部長の片腕、業務知識の深さで信頼を得ている

#### 自社側 1 名

**山田 AE(Account Executive)**
- 30 代後半
- 役割: 価値訴求とヒアリング、SE は同席せず
- 性格: 共感型、相手の言葉を引き出すのが得意

### 4.3 MEDDPICC status マッピング

| # | 項目 | status | 根拠の方向性 |
|---|---|---|---|
| M | Metrics | **partial** | 部長が「予測精度が悪い」と定性的に語り、「年間で数千万くらいの在庫ロスかも」と曖昧な金額を口にする。具体的な ROI 試算はない |
| E | Economic Buyer | **unconfirmed** | 「役員にも相談しないと」と言うが、誰が決裁者か明示されない |
| D | Decision Criteria | **unconfirmed** | 「機能とコストで判断」程度の発言のみ、具体的な評価軸は出ない |
| DP | Decision Process | **unconfirmed** | 「社内で検討します」のみ、稟議フロー・タイムラインは未確認 |
| PP | Paper Process | **unconfirmed** | 契約プロセスの話題は出ない |
| I | Identify Pain | **confirmed** | 課長が具体的な業務フロー・在庫過剰の実例・JIT 納入の精度問題を詳細に語る |
| C1 | Champion | **confirmed** | 部長が「うちの予測精度が悪くて毎月のように工場が振り回されてる、なんとかしたい」と本音を出し、山田 AE に共感を示す |
| C2 | Competition | **unconfirmed** | 競合の話題が出ない(「現状は Excel で凌いでいる」程度) |

**サマリ**: confirmed 2 / partial 1 / unconfirmed 5

### 4.4 起承転結

文字数配分は 5,000〜10,000 字の幅で柔軟に。

#### 起(10〜15%、500〜1,500 字): アイスブレイク + 商談の入口
- 山田 AE が自社製品の概要を 1 分で紹介
- 田中部長が「実は数ヶ月前から需要予測で困ってる」と切り出す
- 雑談を 1 箇所(10〜30 字): 例「先週の展示会、行かれました?」

#### 承(35〜40%、1,800〜4,000 字): 現状ヒアリング → I と C1 を confirmed に
- 鈴木課長が業務フローを詳細に説明
  - 現状: Excel + 営業フォーキャストの手動集計
  - 失敗事例: 「先月、A 部品が欠品して工場停止」「B 部品は過剰在庫で 3 ヶ月分の在庫」
- 田中部長が「これを変えたい」と意思表明 → Champion 性が出る
- 山田 AE が共感しつつ、自社の MEDDPICC ベースのアプローチを紹介
- 業務的な脱線 1 箇所(50〜150 字): 例「ところで、トヨタ系の納入先が増えてるんですよ」(関連はするが本筋ではない)

#### 転(30〜35%、1,500〜3,500 字): 価値訴求 + M を partial に
- 山田 AE が「他社事例では在庫 15% 削減」などの定量例を提示
- 田中部長が「うちなら年間でいくらの効果?」と聞く
- 鈴木課長が「ざっくり計算すると 1〜2 ヶ月の在庫が削れれば…」と試算を始める
- 田中部長「数千万くらいにはなる気はするけど、ちゃんと計算しないと」→ M = partial の evidence

#### 結(15〜20%、800〜2,000 字): 次のステップ
- 山田 AE「次回は ROI シミュレーションを持ってきます」
- 田中部長「うちでも社内で話を進めておきます。役員にも相談しないと」→ E = unconfirmed の evidence
- 鈴木課長「正式な検討は来月以降になるかな」→ DP = unconfirmed の evidence
- 競合の話題は出ずに終わる → C2 = unconfirmed

### 4.5 フィラー・脱線の量と配置ルール

| 種類 | 数 | 配置ルール |
|---|---|---|
| フィラー(えーと、あの、なんていうか) | 5〜10 箇所 | MEDDPICC 評価対象発話の前後に置かない |
| 言い淀み(途中で言い直す) | 3〜5 箇所 | 同上 |
| 軽い雑談(業務に関係ない) | 1 箇所(10〜30 字) | 起のセクションに配置 |
| 業務的な脱線(関連するが本筋ではない) | 1 箇所(50〜150 字) | 承のセクションに配置 |
| 重複発話(同じ内容を別の言い方で繰り返す) | 2〜3 箇所 | どこでも可 |

**理由**: Phase 2 は商談初期 → 顧客側もまだ慣れていない → フィラー多めが自然(現実の商談観察と整合)。

## 5. Phase 4 案件仕様

### 5.1 案件サマリ

| 項目 | 内容 |
|---|---|
| 顧客企業 | Tier 1 自動車部品メーカー(大手、年商 1,000 億超) |
| 取扱製品 | 駆動系部品(例: トランスミッション部品、シャシー部品) |
| 案件規模 | 年間 ¥22M(税抜)、複数拠点導入想定 |
| 商談回数 | 4〜5 回目(最終提案フェーズ) |
| 商談形式 | 顧客先で対面、所要 90 分 |
| 議事録形式 | 録音文字起こし、5,000〜10,000 字 |

### 5.2 登場人物

#### 顧客側 3 名

**高橋 執行役員(製造本部長 兼 執行役員)**
- 52 歳、寡黙だが核心を突く質問をするタイプ
- EB(Economic Buyer)役
- 性格: 余計な言葉を使わない、結論思考
- 立ち位置: 最終意思決定者、投資判断者

**伊藤 部長(情報システム部 部長)**
- 48 歳、慎重派
- DP / PP 管理者 + C2 risk 顕在化役
- 性格: リスクを言語化するのが仕事、SAP IBP との比較を持ち出す
- 立ち位置: 全社視点の整合性担当、契約・調達プロセスの管理者

**渡辺 部長(製造部 部長、Phase 2 の田中部長とは別人物)**
- 46 歳
- Champion 役(過去商談で信頼関係構築済みという前提)
- 性格: 推進力がある、社内根回しに長けている
- 立ち位置: 社内推進役

#### 自社側 2 名

**山田 AE(Phase 2 と同一人物)**
- 同じ Account Executive が両案件を担当している設定
- 役割: 全体ナビゲート、価格・契約条件の交渉

**中村 SE(Solutions Engineer、Phase 4 で初登場)**
- 30 代後半
- 役割: 技術的な質問への回答、SAP IBP との機能比較を語る役
- 性格: 技術詳細に強い、淡々と事実を述べる

### 5.3 MEDDPICC status マッピング

| # | 項目 | status | 根拠の方向性 |
|---|---|---|---|
| M | Metrics | **confirmed** | 高橋執行役員が「在庫削減で年間 ¥1.2 億、機会損失削減で ¥8,000 万、合計 ¥2 億の効果を見込んでいる」と具体的に語る |
| E | Economic Buyer | **confirmed** | 高橋執行役員自身が「私が最終判断する」と明言、稟議は役員会通過済み |
| D | Decision Criteria | **confirmed** | 伊藤部長が評価軸を列挙(機能適合度、TCO、セキュリティ要件、サポート体制、SAP との連携性) |
| DP | Decision Process | **confirmed** | 「来月末までに最終ベンダー選定、再来月から PoC、3 ヶ月後に契約」と明確なスケジュール |
| PP | Paper Process | **confirmed** | 伊藤部長が「契約は弊社のテンプレで、法務レビュー 2 週間、購買稟議 1 週間」と具体的に説明 |
| I | Identify Pain | **confirmed** | 商談冒頭で「前回までの整理として、現状の Pain は X」と振り返る発言を入れる(evidence が今回の議事録内で取れるようにする) |
| C1 | Champion | **confirmed** | 渡辺部長が社内推進役を担っていることが、高橋執行役員の発言から明らか |
| C2 | Competition | **risk** | 伊藤部長が「SAP IBP との比較を本部から求められている。機能では御社が上だが、SAP は全社の ERP と統合できる強みがある」と明示 |

**サマリ**: confirmed 7 / risk 1

### 5.4 起承転結

#### 起(10〜15%、500〜1,500 字): 開会 + 議題確認
- 山田 AE「本日は最終提案ということで、ROI と導入計画を改めて確認させてください」
- 高橋執行役員「時間が限られているので、要点を絞って進めましょう」(寡黙キャラの導入)
- 山田 AE が冒頭で「前回までの整理ですが、現状の Pain は…」と振り返る → I = confirmed の evidence
- フィラー 1〜2 箇所

#### 承(30〜35%、1,500〜3,500 字): M, E, D を confirmed に
- 山田 AE が ROI 試算を提示(¥1.2 億 + ¥8,000 万 = ¥2 億の効果)
- 高橋執行役員「この数字は私も見ている。投資判断としては妥当」→ M, E = confirmed
- 伊藤部長が評価軸を確認(機能、TCO、セキュリティ、サポート、連携)→ D = confirmed
- 業務的な脱線 1 箇所(50〜100 字): 例「ところで、トヨタの新型ハイブリッド向けの部品計画なんですが…」

#### 転(30〜35%、1,500〜3,500 字): C2 で risk 顕在化
- 伊藤部長「正直に申し上げると、本部から SAP IBP との比較を求められている」
- 「機能面では御社の方が需要予測の精度が高いと評価している。ただ、SAP は全社の ERP と統合できる」
- 中村 SE が技術的に反論(「弊社も SAP との API 連携実績は X 社で稼働中」)が、決着はつかない
- 高橋執行役員「この点は次回までに整理が必要」→ C2 = risk の evidence

#### 結(20〜25%、1,000〜2,500 字): DP, PP を confirmed に + 次のステップ
- 伊藤部長が DP / PP を整理(「来月末選定、再来月 PoC、3 ヶ月後契約、契約は弊社テンプレ、法務 2 週間、購買稟議 1 週間」)→ DP, PP = confirmed
- 山田 AE「C2 のリスクに対しては、SAP 連携実績の追加資料を 1 週間以内に提出します」
- 高橋執行役員「では、それを見て最終判断する」

### 5.5 フィラー・脱線の量と配置ルール

| 種類 | 数 | 配置ルール |
|---|---|---|
| フィラー | 2〜5 箇所 | MEDDPICC 評価対象発話の前後に置かない |
| 言い淀み | 1〜3 箇所 | 同上 |
| 軽い雑談 | 0〜1 箇所 | 任意 |
| 業務的な脱線 | 1 箇所(50〜100 字) | 承のセクションに配置 |
| 重複発話 | 1〜2 箇所 | どこでも可 |

**理由**: Phase 4 は商談終盤 → 議論が成熟 → フィラー少なめ、発話が論理的(現実の商談観察と整合)。Phase 2 との対比を作るための意図的な濃度差。

## 6. 出力 JSON 仕様

Day 3 で確定したスキーマを再掲する。

### 6.1 全体構造(フラット 8 項目並列)

```json
{
  "metrics": { "status": "...", "evidence": {...} },
  "economic_buyer": { "status": "...", "evidence": {...} },
  "decision_criteria": { "status": "...", "evidence": {...} },
  "decision_process": { "status": "...", "evidence": {...} },
  "paper_process": { "status": "...", "evidence": {...} },
  "identify_pain": { "status": "...", "evidence": {...} },
  "champion": { "status": "...", "evidence": {...} },
  "competition": { "status": "...", "evidence": {...} }
}
```

### 6.2 status enum

以下の 4 値のみ:

- `confirmed`: 議事録内の発話で十分な根拠がある
- `partial`: 部分的に確認できるが、完全な根拠ではない
- `unconfirmed`: 議事録内で言及されていない、または根拠が不十分
- `risk`: 確認されているが、リスク要因が明示されている

### 6.3 evidence 構造(3 フィールド)

```json
{
  "quote": "発話の引用(クリーンアップ済み、フィラー除去後)",
  "speaker": "話者ラベル(例: 田中 部長(顧客))",
  "interpretation": "なぜこの発話が当該 status の根拠になるかの簡潔な説明"
}
```

### 6.4 quote のクリーンアップ方針

- フィラー(「えーと」「あの」「なんていうか」)を除去
- 言い淀みを除去(言い直し後の最終形を採用)
- 重複発話は最初の発話を採用、または最も明確な言い回しを採用
- 句読点を整える(録音文字起こしの不自然な区切りを修正)
- 内容は変えない(言い換えはしない)

### 6.5 unconfirmed の場合の evidence

unconfirmed の場合、evidence は 2 通りの書き方が可能:

**パターン A**: 該当発話がない場合
```json
{
  "quote": "",
  "speaker": "",
  "interpretation": "議事録内で経済意思決定者(EB)に関する明示的な言及がない"
}
```

**パターン B**: 関連発話はあるが根拠不十分の場合
```json
{
  "quote": "役員にも相談しないと",
  "speaker": "田中 部長(顧客)",
  "interpretation": "決裁者の存在は示唆されているが、誰が EB かは明示されていない"
}
```

Phase 2 では基本的にパターン B を採用する(関連発話がある unconfirmed が多いため)。

## 7. 話者ラベル形式

Day 3 確定 + Day 5 拡張版。

### 形式

```
苗字 役職(顧客/自社):
```

### 例

```
田中 部長(顧客): うちの予測精度が悪くて...
鈴木 課長(顧客): 具体的には、先月もA部品が...
山田 AE(自社): ご事情よくわかります。弊社の事例では...
```

### 拡張理由

Day 3 確定の「役職名(顧客/自社):」では、Phase 4 で「部長」が 2 人(伊藤部長、渡辺部長)出るケースで識別が破綻する。Day 5 で「苗字」を付与する拡張を加えた(可逆な判断のため、軸1 に従い拡張可能)。
