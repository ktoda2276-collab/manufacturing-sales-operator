# Manufacturing Sales Operator (MSO)

> **A Sales-Demand to Manufacturing-Supply Forecasting Bridge**

> *営業の需要予測が、製造の供給計画をつくる。*

---

## このプロジェクトの根底にある問い

**営業の需要予測の精度は、製造の供給計画の精度の天井になる。**

製造業のサプライチェーンは、需要予測から供給計画への変換で動いています。営業マネージャーが「来月3億円受注予定」と言えば、製造は原材料を調達し、ラインを確保し、人員を配置します。

しかし、この営業フォーキャストは、しばしば営業担当者の主観——希望、楽観、忖度、過去の成功体験——によって歪められます。Salesforce社が公表する業界標準では、B2B営業のフォーキャスト的中率は45-50%程度。**営業の予測が外れた瞬間、製造は過剰在庫か機会損失のどちらかを必ず引きます。**

この構造的問題を、生成AIによる商談分析と確率的フォーキャスト集計で解決できないか。それがこのプロジェクトの問いです。

---

## 設計思想：上流の品質を改善することで下流の品質を上げる

このプロジェクトの設計判断は、すべて以下の認知モデルに基づいています。

```
[商談内容文字起こし] → [商談フェーズ判定] → [期待収益算出] → [営業フォーキャスト集計]
   ↑                       ↑                  ↑                ↑
   定性情報                 構造化              定量化            意思決定
   (主観的)                (機械的)           (確率的)         (客観的)

                                                                  ↓
                                                   [製造フォーキャスト連携]
                                                          (製販連携)
```

### この設計が依拠する3つの仮説

**仮説1: 商談の主観バイアスは、定性情報の機械的な判定で軽減できる**

商談担当者に「この商談、何%の確率で受注？」と聞くと、担当者の主観が混ざり需要予測の精度がおちます。代わりに商談内容の文字起こしをMEDDPICC の各項目に基づいて機械的に判定すれば、商談フェーズや受注確率は客観的に導出できる。

**仮説2: LLMにより顧客の言葉による合意形成ができる**

商談内容の文字起こしは自由記述の形式をとりますが、MEDDPICC のような確立されたフレームワークに沿った情報が含まれているはずです。Claudeの構造化能力をつかうことで、顧客の言葉による合意形成という本来の予測にあるべき姿に近づきます。

**仮説3: 営業の需要予測の精度があがれば、製造の供給計画への貢献が可能になる**

約50%という低い需要予測に基づいた供給計画では過剰在庫と機会損失のリスクを常にはらみますが、営業の需要予測の精度があがれば製造側のリスクも自然と下がります。

## ターゲットユーザー

製造業BtoB営業組織のマネージャー層、特に以下の課題を感じている方：

- 部下の営業担当者の予測精度がバラバラで、組織のフォーキャスト精度が低い
- 営業担当者の主観ではなく、顧客の言葉に基づいた合意形成を予測の根拠にしたい
- MEDDPICC や類似の営業方法論を導入したいが、現場定着に苦労している
- 月次フォーキャスト会議が「希望と実績のすり合わせ」になっている
- 営業データを製造側に渡しても「使えない」と言われる

## システム構成（概要）

このプロジェクトは4つのレイヤーで構成されます。各レイヤーはそれぞれ独立して価値を提供し、累積的に統合されます。

```
┌─────────────────────────────────────────────────────────┐
│ Layer 4: Manufacturing Forecast Bridge (設計のみ・将来構想)│
│   - 営業データ → 製造リードタイム接続                     │
│   - 品目別需要予測の生成                                  │
└─────────────────────────────────────────────────────────┘
                          ↑
┌─────────────────────────────────────────────────────────┐
│ Layer 3: Sales Forecast Analytics (v3で実装)             │
│   - 動的確率算出（過去実績ベース）                        │
│   - Well-Architected な設計判断                          │
│   - マルチLLMルーティング（コスト最適化）                  │
└─────────────────────────────────────────────────────────┘
                          ↑
┌─────────────────────────────────────────────────────────┐
│ Layer 2: Enterprise Data Layer (v2で実装)                │
│   - Snowflake 統合                                       │
│   - 過去類似商談 RAG                                     │
│   - Text-to-SQL（自然言語クエリ）                         │
└─────────────────────────────────────────────────────────┘
                          ↑
┌─────────────────────────────────────────────────────────┐
│ Layer 1: MEDDPICC Analysis Engine (v1で実装)             │
│   - 商談内容文字起こし → MEDDPICC 構造化抽出              │
│   - 商談フェーズ自動判定                                  │
│   - 期待収益計算（金額 × 確率）                            │
└─────────────────────────────────────────────────────────┘
```

## v1: MEDDPICC Analysis Engine

### 解決する課題

営業担当者が記録する商談内容文字起こしから、MEDDPICC の8項目について合意状況を機械的に抽出し、商談フェーズと期待収益を客観的に算出する。

### MEDDPICC の8項目（製造業文脈）

| 略号 | 項目 | 製造業BtoBでの具体例 |
|---|---|---|
| **M** | Metrics（数値指標） | 生産性向上◯%、コスト削減◯円など定量目標 |
| **E** | Economic Buyer（経済的決裁者） | 工場長、経営企画、最終予算決裁者 |
| **D** | Decision Criteria（意思決定基準） | 価格、品質、納期、サポートの優先順位 |
| **D** | Decision Process（意思決定プロセス） | 稟議フロー、競合比較、PoC実施可否 |
| **P** | Paper Process（契約プロセス） | 購買部の手続き、契約書フォーマット要件 |
| **I** | Identify Pain（顧客の痛み） | 設備老朽化、人手不足、品質問題 |
| **C** | Champion（社内推進者） | 提案を社内で押してくれる味方 |
| **C** | Competition（競合状況） | 競合他社、内製検討、現状維持 |

### フェーズ判定ロジック

各項目を `confirmed / partial / unconfirmed / risk` の4段階で評価し、フェーズを自動判定する。

```
Phase 1 (Discovery)         : Identify Pain が confirmed
Phase 2 (Qualification)     : + Champion ＋ Economic Buyer
Phase 3 (Solution Dev)      : + Metrics ＋ Decision Criteria
Phase 4 (Negotiation)       : + Decision Process ＋ Paper Process ＋ Competition
Phase 5 (Closing)           : 全8項目 confirmed
```

### 期待収益の算出

```
期待収益 = 商談金額 × 受注確率
受注確率 = フェーズ基本確率 × MEDDPICC確認率の補正
```

v1ではフェーズ別の固定確率（Phase 1: 10%, Phase 2: 30%, Phase 3: 50%, Phase 4: 70%, Phase 5: 90%）を使用。v3で過去実績ベースの動的確率に進化させる。

### 主な機能

- 商談内容文字起こしの入力と保存（SQLite）
- MEDDPICC 8項目の自動抽出（Claude Sonnet 4.6）
- フェーズ自動判定
- 期待収益の自動計算
- 不足項目のハイライトと次のアクション提案
- 商談一覧ビュー（金額・確率・期待収益のソート）
- 評価フレームワーク（30件の自作テストデータセット）

### 技術スタック（v1）

- LLM: Claude Sonnet 4.6 (Anthropic API)
- Backend: Python 3.11+
- Database: SQLite
- Frontend: Streamlit
- Evaluation: pytest + 自作評価データセット

## v2: Enterprise Data Layer

### 解決する課題

v1は単独商談の分析に留まる。組織レベルでフォーキャストを集計し、過去の類似商談を参照する仕組みが必要。

### 主な機能

#### Snowflake 統合
- 商談データを SQLite から Snowflake に移行
- マネージャーが期待収益を業種別・担当者別・月次に集計
- Streamlit ダッシュボードで可視化

#### 過去類似商談 RAG
- Snowflake Cortex AI のベクトル検索機能を活用
- 新規商談内容文字起こしに対して、過去の類似商談 Top3 を提示
- 類似商談の受注成功/失敗を踏まえた戦略提案

#### Text-to-SQL（自然言語クエリ）
- マネージャーが日本語で質問 → Claude が SQL 生成 → Snowflake で実行 → 自然言語で回答
- 例：「今月の業種別フォーキャスト合計は？」「失注した商談で多かった共通項は？」

### 技術スタック（v2追加）

- Snowflake (Cortex AI Functions)
- Snowflake Connector for Python
- 評価ライブラリ（RAG精度、SQL生成精度）

## v3: Sales Forecast Analytics

### 解決する課題

v1の固定確率は「業界平均」でしかない。実際の組織では業種別・金額帯別・季節別に確率が異なる。Production-Ready な実装が必要。

### 主な機能

#### 動的確率算出
- 過去2年分の受注実績データから、業種×金額帯×フェーズ別の動的確率を算出
- SQL集計で算出した確率を v1の固定確率に置き換え

#### マルチLLMルーティング
- 簡易タスク → Claude Haiku 4.5（高速・低コスト）
- 標準タスク → Claude Sonnet 4.6（バランス）
- 複雑タスク → Claude Opus 4.7（最高精度）
- タスク種別による自動ルーティング

#### Well-Architected な設計判断のドキュメント化

AWS Well-Architected Framework の6つの柱に沿った設計判断を `docs/well_architected.md` にまとめる：
- Reliability（リトライ機構、フォールバック戦略）
- Security（APIキー管理、SQL injection対策）
- Cost Optimization（モデルルーティング、キャッシング）
- Performance Efficiency（並列処理、レスポンスタイム最適化）
- Operational Excellence（ログ、メトリクス、評価の自動化）
- Sustainability（不要リソースの削減）

## v4 (Future): Manufacturing Forecast Bridge

### 解決する課題

ここまでで構築された営業フォーキャストデータを、製造側の供給計画に接続する。これがプロジェクトの究極の問いに答える層。

### 設計コンセプト（実装はせず、設計ドキュメントのみ）

```
営業データ:                       変換ロジック:                製造データ:
deals                             Sales-to-Production         production_plan
- customer                        Bridge                       - product_code
- amount             ────►        - 商品マスタとの紐付け  ────►   - quantity
- expected_close_date             - リードタイム計算           - manufacturing_start
- phase / probability             - 期待値ベースの数量変換     - capacity_check
- product_category                                            
```

### 検討するべき設計判断

- 確率重み付けの考え方：期待値で計算するか、シナリオ別（楽観/標準/悲観）で計算するか
- リードタイムの取扱い：商品ごとに異なるリードタイムをどう扱うか
- 製造キャパシティとの突合：実現困難な需要をどう営業側にフィードバックするか
- 更新頻度：日次/週次/月次のどれが現実的か

### このプロジェクトでの取扱い

実装は行わず、`docs/manufacturing_bridge_design.md` で以下を論じる：
- アーキテクチャ図
- データモデル設計
- 想定されるエッジケース
- 既存ERP（SAP S/4HANA等）との接続パターン

実装しない理由は「個人プロジェクトのスコープを超える」と「設計だけで価値を伝えられる」の両方。

## 評価フレームワーク

このプロジェクトは「動くデモ」ではなく「精度を測れる実装」を目指します。

### v1 の評価指標

- MEDDPICC項目判定精度（8項目 × 30件 = 240判定の正解率）
- フェーズ判定精度（30件中の正解率）
- 確率算出の妥当性（人間の感覚値とのズレ）

### v2 の評価指標

- RAG 検索精度（類似商談の関連性スコア）
- Text-to-SQL 生成精度（生成SQLの実行成功率と意図一致率）

### v3 の評価指標

- 動的確率の予測精度（過去データでの的中率）
- マルチLLMルーティングのコスト削減率
- レスポンスタイム

## 設計上の判断と意図的な制約

INTP的な性質として、このプロジェクトには以下の意図的制約を置きます。

### 意図的に「やらない」こと

- **本物のSalesforce連携**: 現職のSalesforceを使った課題解決の思考に運用が寄ってしまうのを避けるため。本プロジェクトは特定CRMベンダーに依存しない、汎用的な営業フォーキャスト基盤としての設計を意図する。
- **本物の企業データ**: 全データは架空、コンプライアンスリスクを排除
- **製造システムの実装**: 製造システム（＝基幹システム）は要件が複雑で個人での開発には向かないため。ただし、製造システムとの連携方法については設計ドキュメント（`docs/manufacturing_bridge_design.md`）で論じる。
- **多言語対応**: 日本語のみ、本質に集中
- **モバイル対応**: デスクトップのみ、本質に集中

### なぜこれらを「やらない」のか

SAの仕事の本質は「何をやらないか」を判断すること。「全部やる」と表明することは、ジュニアの発想であって、SA思考ではない。意図的な制約を明示することは、設計判断ができることの証明です。

## なぜこのプロジェクトを作るのか

### 著者の動機

製造業の顧客への営業現場で「フォーキャストが当たらない」という声を10年以上聞いてきました。原因を分解すると、データの問題というより**営業組織の認知の問題**であることが多かったように思います。

「うちの営業マネージャーは予測が下手」というより、「予測を客観的に行う構造がない」のです。営業担当者が「来月は確実」と言えば、組織はその主観を集計するしかない。**営業担当者の言葉ではなく、顧客の言葉を予測の根拠にする仕組み**が、構造的に欠けていました。

生成AIの登場で、商談内容文字起こしという定性情報を構造化できるようになりました。これは**営業組織の認知のアップグレード**を意味します。それを実装で示したい——これがこのプロジェクトの個人的動機です。

### 技術的に学びたいこと

- LLMによる構造化情報抽出の限界と可能性
- エンタープライズデータ基盤（Snowflake）でのRAG実装
- Text-to-SQL の精度をどこまで上げられるか
- Well-Architected な AI アプリケーションとは何か
- 営業データから製造データへの変換ロジックの設計
