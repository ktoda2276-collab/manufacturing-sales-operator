## 2. 設計判断軸の進化

Day 2-3 で 3 軸を確立し、Day 5 で 4 軸目を追加した。

### 既存の 3 軸(Day 2-3 で確立)

| 軸 | 内容 | 確立 Day |
|---|---|---|
| 軸1 | 可逆な判断には YAGNI、不可逆な判断には先回り設計 | Day 2 |
| 軸2 | 判断は LLM、計算は Python(関心の分離の LLM 時代版) | Day 3 |
| 軸3 | Defense in Depth(各層が独立に防御、全層同時にすり抜ける確率を実用上ゼロまで下げる) | Day 3 |

### Day 5 で追加した軸4

**軸4: Few-shot は「Phase の幅」と「status の書き方の対比」を示す教材である**

詳細:
- Phase 2 と Phase 4 は **対比として機能する**ことが最重要
- 各項目の status(confirmed/partial/unconfirmed/risk)の書き方が **明示的に異なる**ことを示す
- 業種・登場人物・金額などは「対比を成立させる範囲で」決める(こだわりすぎない)

### 軸4 を最初に確立したことの効果

論点 A〜E すべてが、軸4 への適合度で機械的に判断できた。

例:
- 「業種を Phase 2 と Phase 4 で同じにすべきか?」→ 軸4 から「対比は Phase でのみ作るべき」→ 同じ業種が良い
- 「金額を Phase 2 と Phase 4 で同じにすべきか?」→ 軸4 から「金額が違えば M の書き方の対比も作れる」→ 違う金額が良い
- 「フィラーを入れるべきか?」→ 軸4 から「対比を邪魔しない範囲なら入れて OK、Phase 間で量の差を作れば対比強化」→ 適度に入れる

### Day 4 で克服した罠との接続

Day 4 までで「設計議論で永遠に迷う罠」は克服していた(可逆/不可逆軸が機能)。Day 5 では **新しいレベルの判断軸**(Few-shot の役割定義)を追加することで、さらに高速に判断できた。

> **学び**: 設計判断軸は固定ではなく、議論の対象が変わるたびに、その対象に固有の軸を 1 本追加すれば、判断速度が一気に上がる。

これは Day 5 の最も重要な学びかもしれない。

## 3. 論点 A〜E の議論ログ

### 論点A: 業種選定

#### 検討した選択肢
1. 自動車部品メーカー向け生産管理システム
2. 食品メーカー向け需要予測ツール
3. 化学メーカー向け品質管理システム

#### 確定: 自動車部品×生産計画 SaaS(同じ業種、別顧客)

#### 判断根拠
- **軸4 適用**: 業種を 2 種類にすると「Phase の違い」と「業種の違い」が混ざって、Claude が何を学ぶべきか分散する。同じ業種に揃えるべき
- **MSO ドメインとの整合**: 自動車部品×生産計画は CLAUDE.md §1 の設計思想「営業の需要予測が製造の供給予測をつくる」と完全に重なる。Few-shot 自体が MSO の存在理由の例示になる
- **執筆コスト**: 業界用語・登場人物の役職・システム要件を 1 セットで済む

#### 「同じ顧客 vs 別顧客」での追加判断

選択肢:
- 同じ顧客の Phase 2 → Phase 4 の進化を見せる
- 別顧客の異なる Phase を独立に見せる ← **採用**

採用理由:
- Few-shot として「Phase は時系列で進む」と誤学習するリスクを避けたい
- 登場人物・組織規模・競合状況を独立に設計できる自由度が必要
- Phase 4 の C2 risk(SAP IBP との比較)は、別顧客の方が状況設定が自由

### 論点B: 商談規模

#### 検討した選択肢
- 同程度の金額にする
- 違う金額にする ← **採用**

#### 確定
- Phase 2 = ¥8M / 年(中型、Tier 2)
- Phase 4 = ¥22M / 年(大型、Tier 1)
- 金額差: 約 2.75 倍

#### 判断根拠
- **軸4 適用**: 金額が違うと、M (Metrics) の書き方も自然に変わる
  - 小型: ROI 根拠が単純(「在庫削減で年間 X 円」)
  - 大型: ROI 根拠が複合的(「在庫 + 機会損失 + 稼働率 + 交渉力」)
  - これが Phase 2 の M (partial) と Phase 4 の M (confirmed) の対比を強化する
- **金額帯の意味**:
  - ¥8M = 部長決裁 + 役員報告の境界 → Phase 2 の「予算未確認」が自然
  - ¥22M = SAP IBP の射程に入る → Phase 4 の C2 risk が自然に発生

### 論点C: 登場人物

#### 確定
| | Phase 2 | Phase 4 |
|---|---|---|
| 顧客側 | 2 名(部長 + 課長) | 3 名(執行役員 + 部長 × 2) |
| 自社側 | 1 名(AE のみ) | 2 名(AE + SE) |
| 経営層 | 出さない(話題のみ) | 出す(EB 役) |

#### 判断根拠
- **軸4 適用**: Phase によって登場人物が変わるのが MEDDPICC の自然な姿
  - Phase 2 で経営層を出すと、EB が confirmed になってしまう → unconfirmed を表現するためには「話題に出るが登場しない」状態が必要
  - Phase 4 で執行役員 + 情シス部長を出すことで、EB と DP/PP の confirmed が自然に表現できる
- **競合(C2 risk)の文脈**: 情シス部長を登場させることで、「SAP IBP との比較」の発話が自然に出る

#### ペルソナ濃度の判断

選択肢:
- 薄い(役職のみ)
- 中程度(年齢 + 性格特徴 1 つ + 立ち位置)← **採用**
- 濃い(経歴 + 口癖 + 私生活)

採用理由: 薄すぎると発話が単調、濃すぎると Few-shot が長くなりすぎてコスト増・集中力分散。中程度なら口調や立場が発話に滲む。

### 論点D: 起承転結

#### 確定
- Phase 2: I/C1 confirmed + M partial + 他 unconfirmed
- Phase 4: 7 項目 confirmed + C2 risk
- Phase 4 冒頭で I を軽く振り返る(evidence が今回の議事録内で取れるようにする)

#### 判断根拠
- Day 3 の論点 1C(Few-shot 2 件のフェーズ選定)で確定した内容を、起承転結の具体構造に落とした
- I を Phase 4 冒頭で振り返るのは、「全 8 項目の evidence が今回の議事録内で見つかる」状態を作るため → Claude の学習が綺麗になる

### 論点E: フィラー・脱線

#### 確定
- 適度に入れる
- Phase 2 多め、Phase 4 少なめ(成熟度の対比)
- evidence の quote はクリーンアップ済み(フィラー除去後)
- MEDDPICC 評価対象発話の前後にフィラー・脱線を入れない

#### 判断根拠
- **入力ソースとの整合**: Day 3 で「入力 = 録音文字起こし、フィラー対応は Claude にプロンプト指示で委ねる」と決めた。Few-shot にフィラーがゼロだと、プロンプトの指示「フィラーを無視せよ」が実例で示されない
- **軸4 適用**: Phase 2 と Phase 4 でフィラー濃度を変えると、対比が豊かになる(成熟度の差として読み取れる)
- **クリーンアップ済み quote の意義**: Few-shot でクリーンアップ後の quote を見せることで、「フィラーは除去せよ」というメッセージが暗黙に伝わる(Defense in Depth の一層として機能)

## 4. 論点C の追加判断

### 4.1 苗字付与

#### 問題
Phase 4 で「部長」が 2 人いる(伊藤部長、渡辺部長)。Day 3 確定の話者ラベル形式「役職名(顧客/自社):」では識別が破綻する。

#### 判断根拠

- **実在人物リスク**: フルネームは避ける(偶然の一致リスク)
- **形式の最小限の拡張**: 「苗字 役職(顧客/自社):」とすることで、Day 3 の趣旨を保ったまま識別性を確保

### 4.2 AE 同一人物化

#### 確定: 同じ人物(山田 AE が両案件を担当)

- **面接ナラティブ**: 「Few-shot は別顧客 2 件だが、同じ AE が担当している設定にした。理由は MSO の実運用文脈と整合させるため」と語れる

## 5. SPEC.md 執筆段階で発見した追加論点

SPEC.md を書く過程で、Day 5 の論点 A〜E では議論していなかった論点が浮上した。

### 5.1 unconfirmed evidence のパターン A/B

#### 問題
Day 3 で evidence は「quote + speaker + interpretation」の 3 フィールドと確定したが、unconfirmed の場合の書き方が未確定だった。

#### 2 つのパターン

**パターン A**: 該当発話がない場合
```json
{
  "quote": "",
  "speaker": "",
  "interpretation": "議事録内で経済意思決定者(EB)に関する明示的な言及がない"
}
```

**パターン B**: 関連発話はあるが根拠不十分の場合
```json
{
  "quote": "役員にも相談しないと",
  "speaker": "田中 部長(顧客)",
  "interpretation": "決裁者の存在は示唆されているが、誰が EB かは明示されていない"
}
```

#### 確定: Phase 2 では基本的にパターン B を採用

#### 判断根拠
- Phase 2 のシナリオでは、unconfirmed 項目でも関連発話が多い(部長が「役員にも相談」と言うなど)
- パターン B の方が、Claude に「unconfirmed でも根拠を探せ」というメッセージが伝わる
- 完全に発話がない項目はパターン A(議事録から該当発話を探さない/作らない方針)

#### この論点の位置づけ

これは Day 3 で予見されていなかった論点。**Few-shot を実際に作る段階で初めて顕在化する設計判断** がある、という学びになる。

> **学び**: 設計と実装の境界で、新しい判断が必要になる場合がある。Day 5 では SPEC.md という中間成果物を作る過程で、Day 3 の決定事項を補完する判断が複数発生した。

### 5.2 自社社名の扱い

#### 問題
Few-shot 議事録の中で「弊社では…」「御社の…」と呼び合うが、自社の社名を明示するか?

#### 暫定: 明示しない、「弊社」で通す

#### 判断根拠
- 社名を明示すると、実在企業との混同リスク
- 「弊社」で通せば、Few-shot 内では問題にならない
- ただし、Claude Code が議事録を書く段階で「これだと不自然」となれば判断見直し

#### 残された判断

Claude Code への生成依頼時に、「社名は明示せず『弊社』で通すこと」を明示的に指示する必要がある。SPEC.md の §3 共通設定にメモはあるが、§8 の依頼テンプレートには反映されていない。Day 6 で生成依頼する前に、依頼テンプレートを更新するか口頭で補足するかを判断する。

## 6. 副次的な学び(議論プロセスの振り返り)

### 6.1 設計判断軸を最初に確立する効果

Day 5 の冒頭で軸4(Few-shot の役割定義)を確立してから論点 A〜E に入った。これにより、各論点で 5〜10 分で判断できた。

もし軸4 を確立せずに論点 A から議論していたら、「業種を何にするか」で 30 分以上迷っていた可能性がある。INTP の「設計議論で永遠に迷う罠」を、判断軸の事前確立で回避できた。

### 6.2 「中間成果物を作る過程で新しい判断が顕在化する」現象

§5 で書いた通り、SPEC.md 執筆中に Day 3 では予見されていなかった判断(unconfirmed evidence のパターン、社名の扱い)が浮上した。

これは設計と実装の境界で起きる典型的な現象。完璧な設計を最初から作ろうとせず、**中間成果物を作りながら判断を補完していく** アプローチが現実的。

### 6.3 ペルソナ濃度のスイートスポット

ペルソナ濃度を「中程度」に確定したのは Day 5 の判断だが、これは Few-shot 全体の長さ管理にも繋がる。

- 薄すぎる: 発話の個性が出ない → status 判定の根拠として弱い
- 濃すぎる: Few-shot が長くなる → コスト増 + Claude の集中力分散(Day 3 論点 1C と同じ罠)

「中程度」のスイートスポットを言語化できたのは、軸4 の「対比を成立させる範囲で」という制約条件のおかげ。

## 7. 面接ナラティブ素材

Day 5 で発生した、面接で語れる素材を抽出する。

### 7.1 「設計判断軸を 1 本追加するだけで、後続の判断速度が一気に上がる」

> Few-shot の仕様議論を始める前に、『Few-shot の役割定義』を判断軸として明文化しました。これによって、業種選定・金額設定・登場人物配置などの個別論点が、すべて『この役割定義に貢献するか?』で機械的に判断できるようになり、5 論点を 60 分で確定できました。

### 7.2 「Few-shot の主目的は『Phase の幅』と『status の対比』」

> MEDDPICC 抽出の Few-shot は 2 件作りました。Phase 2 と Phase 4 を選んだ理由は、商談中盤(未確定多め)と終盤(ほぼ確定 + 一部リスク)の対比を見せることで、中間 Phase も推測可能にするためです。1 件だと特定パターンの過学習、5 件以上だとコストと集中力分散で逆効果になります。

### 7.3 「同じ業種・別顧客・別 AE」の意図設計

> Few-shot 2 件は同じ業種(自動車部品×生産計画 SaaS)にしました。業種を揃えることで『Phase の違い』と『業種の違い』が混ざらず、Claude が学ぶべき変数が 1 つに絞られます。一方で顧客企業は別にしました。これは『Phase は時系列で進む』という誤学習を避けるためです。

### 7.4 「設計と実装の境界で新しい判断が顕在化する」

> 完璧な設計を最初から作ろうとせず、中間成果物を作りながら判断を補完するアプローチを採用しました。実際、仕様書を書く過程で『unconfirmed の場合の evidence の書き方』という Day 3 では予見されていなかった論点が浮上し、その場で決定しました。これは要件定義と詳細設計の境界で起きる典型的な現象で、現場で柔軟に判断する設計プロセスです。

### 7.5 「Defense in Depth の Few-shot レベルへの拡張」

> Few-shot の evidence quote をクリーンアップ済み(フィラー除去後)にしました。これにより、プロンプトの『フィラーを無視せよ』という指示と、Few-shot の実例が一致します。プロンプトレベル(指示)と Few-shot レベル(実例)の二層で同じメッセージを伝えることで、Claude のフィラー除去が確実になります。これは Day 3 で確立した Defense in Depth 軸の Few-shot レベルへの拡張です。
